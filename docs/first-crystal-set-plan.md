# First Crystal Set Plan

## Purpose

This draft defines the first minimal crystal set needed for the next system iteration.

The project has now reached a point where one single crystal is not enough to carry all major structures.
A first iteration should therefore use a small crystal cluster rather than one overloaded crystal.

---

## Core Judgment

The next system iteration should not depend on one giant crystal.
It should begin with:

## a minimal crystal set

so that different kinds of structure can be separated cleanly.

Why:
- system architecture is not the same as methodology
- methodology is not the same as runtime retrieval orchestration
- calibration requirements should not be mixed blindly with every structural layer

---

## Recommended First Crystal Set

### Crystal A. System Architecture Crystal
**Working name:** `system-memory-crystal-v1`

### Role
Carry the structural architecture of the memory system itself.

### Should contain
- memory substrate
- ingest / retrieval loop
- meditation / experience engine
- metacognition / governance / feedback
- knowledge crystal layer
- transfer layer
- system-level tensions
- system-level unresolved conflicts
- major runtime paths

### Main question
Can another agent regrow the same system neighborhood from this crystal?

### Current status
Already prototyped.

---

### Crystal B. Methodology Crystal
**Working name:** `methodology-crystal-v1`

### Role
Carry the method by which the system should discover and adapt rules,
not the system architecture itself.

### Should contain
- Rule Discovery Bench logic
- self-definition through test obligations
- content core vs self-adaptation boundary
- pre-ingestion calibration
- architecture regrowth fidelity vs implementation regeneration fidelity
- transfer content core / rebuild adaptation layer
- anti-hardcoding boundary

### Main question
Can another agent understand how to self-calibrate and derive its own unfold / reconstruction policy,
rather than just reading static structure?

### Current status
Not yet crystalized as a formal object.

---

### Crystal C. Retrieval Routing Crystal
**Working name:** `retrieval-routing-crystal-v1`

### Role
Carry the logic by which runtime prompt construction should route across layers.

### Should contain
- memory retrieval mode
- experience retrieval mode
- crystal retrieval mode
- transfer / regeneration retrieval mode
- routing rules by task type
- packaging order
- layer permissions / exclusions
- anti-flat-top-k assumptions

### Main question
Can the runtime system know which layer to retrieve from,
and in what order, rather than using one flat retrieval mechanism?

### Current status
Conceptually clear, not yet crystalized.

---

## Why Three Crystals First

These three are the minimum useful separation because they answer three different questions:

### System Architecture Crystal
What is the system?

### Methodology Crystal
How should an agent discover the right way to use and rebuild it?

### Retrieval Routing Crystal
How should the runtime system call different layers when building prompts or regrowth context?

Putting all three into one crystal would create boundary confusion.

---

## Expected Crystal Relationships

### A -> B relationship
The system architecture crystal should be interpreted through the methodology crystal.

Meaning:
- a receiving agent should not unfold the system crystal blindly
- the methodology crystal helps define calibration obligations and adaptation behavior

### B -> C relationship
The methodology crystal should constrain how retrieval routing is designed.

Meaning:
- routing should respect self-discovery and calibration obligations
- routing should not flatten all retrieval into one static query path

### C -> A relationship
The retrieval routing crystal determines how system architecture knowledge is called into runtime contexts.

Meaning:
- the runtime system needs a routing policy for when to retrieve memory, experience, or crystal-layer structures

---

## First Iteration Scope

The first crystal set should remain small.

Do not attempt yet:
- full crystal ecology
- all subdomains at once
- exhaustive transfer formats
- automatic crystal generation for every concept

Instead:
- build a minimal coherent crystal cluster
- test whether agents can use it to calibrate and regrow system structure

## Current Principle

The first system iteration needs:

## not one total crystal,
## but a small crystal cluster with distinct responsibilities

This is the minimum stable path toward a real crystal-based system.
