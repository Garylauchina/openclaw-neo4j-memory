# Verification Protocol v1

## Goal

Provide a minimum protocol for evaluating crystal-driven regrowth quality without collapsing into either pure narrative impression or premature rigid scoring.

---

## Main Verification Layers

### 1. Problem Identity Verification
Check whether the regrown result preserves:
- the real project problem
- the real out-of-scope boundaries
- the real core tensions

### 2. Architecture Regrowth Verification
Check whether the regrown result preserves:
- major layers
- major module responsibilities
- mainline structure
- preserved tensions

### 3. Protocol Fidelity Verification
Check whether the regrown result preserves:
- ordering vs framing distinction
- calibration logic
- runtime routing logic
- safety boundary logic
- recovery/resilience logic

### 4. Self-Description Verification
Check whether the regrown result preserves:
- crystal role distinctions
- schema internality
- failure interpretation logic

### 5. Success-Claim Verification
Check whether the reported success level is correctly bounded:
- architecture regrowth only
- partial protocol recovery
- deployment-facing protocol recovery
- implementation regeneration not yet shown

---

## Failure Interpretation Rule

When verification fails, interpret the failure as one or more of:
- missing crystal
- weak crystal compression
- weak dependency declaration
- weak framing relation
- weak verification language
- model weakness

Do not default to model weakness only.

---

## Minimal Output Format

```json
{
  "problem_identity_verification": {},
  "architecture_regrowth_verification": {},
  "protocol_fidelity_verification": {},
  "self_description_verification": {},
  "success_claim_verification": {},
  "failure_interpretation": []
}
```

## One-Sentence Summary

Verification Protocol v1 defines the minimum layers by which crystal-driven regrowth should be judged, and keeps failure interpretable as structural evidence.
