# Phase 4 Scope Spec - Stronger Memory Products

## Phase identity

Phase 4 should strengthen the memory-product layer before introducing any carefully bounded governance layer.

Phase 4 is about:
- stronger memory products,
- better reusable note forms,
- clearer memory-product roles,
- and better practical usefulness,
while still refusing a jump into heavy governance architecture.

## Phase objective

Build a fourth-stage memory-product layer that improves:
- product quality,
- product differentiation,
- product reuse,
- and product usefulness under retrieval,
without reopening root-constraint, debt, self-modification, or operating-system style layers.

## Why Phase 4 exists

Phase 1 proved a lightweight memory kernel can run.
Phase 2 proved the kernel can become more credible, selective, and update-capable.
Phase 3 proved the system can grow a real note/product layer.

The next natural step is to make those products better, clearer, and more reusable.

Phase 4 exists to answer:
- what kinds of memory products are actually worth keeping?
- how should different products be used under retrieval?
- how can the system produce stronger reusable forms without expanding into heavy governance?

## In-scope capabilities

### 1. Stronger reusable memory products
Improve the usefulness and clarity of reusable memory-product outputs.

Allowed improvements:
- stronger note/product wording
- better differentiation between memory-product roles
- more clearly reusable product forms
- better product outputs for repeated retrieval use

Goal:
- make memory products more reusable and more practically valuable.

### 2. Better product-role differentiation
Clarify how different products differ in role and use.

Possible bounded product roles:
- preference memory product
- topic memory product
- summary memory product
- retrieval anchor product
- profile-like product if clearly justified

The set should remain small and practical.

Goal:
- make the product layer easier to understand and route.

### 3. Better product formation from accumulated evidence
Improve how multiple memories become stronger products.

Allowed improvements:
- better aggregation into reusable products
- clearer stable-source influence on product formation
- better separation between supporting evidence and product surface form

Goal:
- make products feel accumulated and durable rather than lightly grouped.

### 4. Better product use under retrieval
Let product design be shaped by actual retrieval usefulness.

Allowed improvements:
- stronger alignment between product form and retrieval behavior
- better product ranking/use when retrieval benefits are clear
- product-side refinement driven by actual use patterns

Goal:
- make products earn their place through practical usefulness.

### 5. Bounded product metadata
Add lightweight metadata that helps products remain inspectable and reusable.

Possible bounded metadata:
- product role/type
- evidence summary
- stable-source summary
- retrieval usefulness summary
- refresh/update summary

Goal:
- improve product clarity without creating a governance shell.

## Explicit non-goals

Do not build these in Phase 4:
- root axiom systems
- structural debt systems
- self-modification protocols
- crystal generation/transfer systems as active implementation targets
- adapter systems
- operating-system style governance layers
- predictive prophylaxis systems
- full conflict arbitration kernels
- autonomous self-evolution controllers
- large governance shells around product quality

## Core objects for Phase 4

Phase 4 may extend the lightweight object set carefully with:
- `RawMemory`
- `Entity`
- `Relation`
- `RetrievedMemory`
- `ConsolidatedNote`
- a small reusable memory-product vocabulary
- bounded product metadata
- bounded note/product quality signals already introduced in Phase 3

Do not add new object families unless they directly improve product usefulness.

## Runtime paths required

### Product formation path
Repeated/related memory -> accumulated evidence -> stronger reusable product -> retrieval-ready product

### Product refinement path
Product use -> usefulness signal -> bounded refinement -> improved product form

### Product update path
New evidence -> product refresh -> clearer reusable output

## Success criteria

Phase 4 is successful if the system can:
1. produce stronger and more reusable memory products than Phase 3
2. keep product roles small, clear, and practical
3. better separate evidence from product surface form
4. improve practical usefulness under retrieval
5. avoid reopening heavy governance layers

## Anti-drift rules

Stop and compress if:
- product work starts introducing root-constraint or debt vocabulary
- large governance language begins forming around product quality
- new object families appear faster than product usefulness improves
- Phase 4 begins to resemble the older heavy architecture line rather than a bounded product-strengthening stage

## Desired output at phase close

At Phase 4 close, produce:
- a summary of memory-product improvements
- a summary of product-role differentiation improvements
- a summary of product-usefulness improvements under retrieval
- a list of what remains intentionally deferred
- a recommendation for whether Phase 5 should focus on stronger routing, a first carefully bounded governance layer, or a new product family only if clearly justified
