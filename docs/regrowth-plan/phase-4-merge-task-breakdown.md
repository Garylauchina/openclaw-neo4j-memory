# Phase 4 Merge Task Breakdown

This file turns Phase 4 into small mergeable steps.

Phase 4 target:
- stronger memory products
- better reusable note/product forms
- better product-role differentiation
- better practical usefulness
- while preserving the lightweight regrowth line

## Merge 1 - Stronger reusable memory products

### Goal
Make memory products more reusable and more practically valuable than in Phase 3.

### Required implementation
- improve product wording and product structure
- improve product outputs for repeated use
- make products more reusable than Phase 3 notes
- ensure products better reflect accumulated evidence

### Required decisions
- what makes a memory product stronger in this phase?
- how much wording and structure improvement is useful before it becomes decorative?
- what reusable surface forms are worth introducing?

### Required verification
- compare product outputs against Phase 3 notes on representative examples
- confirm products are more reusable and clearer
- confirm improvements remain bounded and practical

### Merge gate
Merge only if product quality visibly improves without introducing product-side governance machinery.

## Merge 2 - Better product-role differentiation

### Goal
Clarify a small but useful set of product roles.

### Required implementation
- define a bounded product-role set
- support some of:
  - preference product
  - topic product
  - summary product
  - retrieval anchor product
  - profile-like product only if clearly justified
- route product formation into these roles consistently

### Required decisions
- which product roles are truly useful in Phase 4?
- which roles would be premature or taxonomy-heavy?
- how should product-role selection remain inspectable?

### Required verification
- confirm multiple product roles can be produced consistently
- confirm the role set remains small and practical
- confirm role differentiation improves reuse and retrieval behavior

### Merge gate
Merge only if product-role differentiation increases usefulness without taxonomy inflation.

## Merge 3 - Better product formation from accumulated evidence

### Goal
Make accumulated evidence form stronger products than the current Phase 3 layer.

### Required implementation
- better aggregate multiple memories into stronger products
- clarify the influence of stable sources on product formation
- better separate supporting evidence from the reusable product surface

### Required decisions
- how much stable evidence should shape product surface form?
- when should a product be strengthened versus reformulated?
- how much evidence detail should remain visible in the product?

### Required verification
- compare accumulated-evidence product formation against Phase 3 behavior
- confirm products become more durable and more representative
- confirm the mechanism remains lightweight and inspectable

### Merge gate
Merge only if accumulated evidence improves product quality without opening a heavy evidence/governance layer.

## Merge 4 - Better product use under retrieval

### Goal
Make stronger products matter more under practical retrieval.

### Required implementation
- improve product-side retrieval usefulness
- better align product form with retrieval behavior
- let retrieval usage shape bounded product refinement
- make useful products easier to reuse without overdominating raw memory

### Required decisions
- what counts as a product being useful under retrieval in this phase?
- how much product promotion is helpful before routing becomes too rigid?
- how should product usefulness remain inspectable?

### Required verification
- confirm product usefulness can influence retrieval behavior
- confirm stronger products are more likely to be reused
- confirm raw memory remains available and the system does not collapse into product-only retrieval

### Merge gate
Merge only if retrieval usefulness improves without collapsing retrieval diversity.

## Merge 5 - Bounded product metadata

### Goal
Add lightweight metadata that helps products remain inspectable and reusable.

### Required implementation
- introduce a bounded set of product metadata, such as:
  - product role/type
  - evidence summary
  - stable-source summary
  - retrieval usefulness summary
  - refresh/update summary
- use metadata only to support product clarity, reuse, and bounded prioritization
- keep metadata descriptive and practical

### Required decisions
- which metadata is truly useful in Phase 4?
- which metadata would be decorative or governance-like?
- how should metadata remain visible and understandable?

### Required verification
- confirm metadata improves inspection and reuse
- confirm metadata does not expand into a heavy product-governance shell
- confirm product clarity improves in visible ways

### Merge gate
Merge only if metadata improves product clarity while remaining bounded and practical.

## Cross-merge anti-drift rules

Do not allow these to enter Phase 4 merges:
- root axiom systems
- structural debt systems
- self-modification protocols
- crystal/transfer implementation layers
- operating-system style governance layers
- predictive prophylaxis systems
- full conflict arbitration kernels
- autonomous self-evolution controllers
- large governance shells around product quality

If any of these appear, stop and compress before continuing.

## End-of-phase closeout

At the end of Phase 4, produce a closeout note answering:
- how memory products improved relative to Phase 3
- how product-role differentiation improved
- how product usefulness improved under retrieval
- what remains intentionally deferred
- whether Phase 5 should focus on stronger routing, a first carefully bounded governance layer, or a new product family only if clearly justified
