# Phase 4 Working Boundary

This file defines what Phase 4 should actively improve, what it should use only as reference, and what it must continue to defer.

## Purpose

Protect Phase 4 from turning memory-product strengthening into a heavy governance or abstraction layer.

## Keep in active focus

### Planning docs
- `docs/regrowth-plan/phase-3-closeout-note.md`
- `docs/regrowth-plan/phase-3-freeze-summary.md`
- `docs/regrowth-plan/phase-4-scope-spec.md`
- `docs/regrowth-plan/phase-4-merge-task-breakdown.md`
- `docs/regrowth-plan/phase-4-working-boundary.md`

### Code concerns that belong to Phase 4
- stronger reusable memory products
- better product-role differentiation
- better product formation from accumulated evidence
- better product use under retrieval
- bounded product metadata

### Data/object concerns that belong to Phase 4
- `ConsolidatedNote`
- small reusable memory-product vocabulary
- bounded product metadata
- accumulated-evidence product formation behavior
- retrieval-shaped product refinement behavior

## Use only as reference/baseline

### The older heavy line
Treat the architecture/crystal/governance line as reference only.
Use it for contrast, not as immediate scope.

### Why reference only
Phase 4 is still a bounded product-strengthening phase.
It should not absorb crystal, axiom, debt, or operating-system architecture ideas as direct implementation targets.

## Explicitly ignore in Phase 4 implementation

Do not actively implement or reintroduce these in Phase 4:
- root axiom systems
- structural debt systems
- self-modification protocols
- crystal generation/transfer implementation
- adapter systems
- operating-system style governance layers
- predictive prophylaxis systems
- full conflict arbitration kernels
- autonomous self-evolution controllers
- large scoring economies around product quality

## Defer until Phase 5 or later

### Possible Phase 5 topics
- stronger routing if directly justified
- the first carefully bounded governance layer
- a new product family only if clearly justified

### Later than Phase 5 unless strongly justified
- root-constraint / axiom systems
- debt/pruning economies
- self-evolution control loops
- crystal-transfer implementation layers
- operating-system style architectural metaphors as implementation drivers

## Practical working rule

When making a Phase 4 change, ask:
1. Does this improve the usefulness of memory products?
2. Does this improve product formation, product reuse, or product clarity?
3. Could this be postponed without harming the Phase 4 target?

If the answer to (3) is yes, postpone it.

## Merge hygiene rule

A Phase 4 merge should mostly touch:
- minimal implementation files in the lightweight kernel line
- regrowth-plan docs
- minimal verification artifacts

A Phase 4 merge should not become a broad abstraction or vocabulary expansion sweep.

## Compression trigger

Stop and compress before continuing if:
- governance or ontology vocabulary grows faster than product usefulness improves
- new object families appear without improving product reuse
- implementation starts depending on root-constraint, debt, or self-modification abstractions
- Phase 4 begins to resemble the older heavy line instead of a bounded product-strengthening stage
