# Three-Crystal Runtime Routing Self-Experiment v1

## Purpose

This document records the first three-crystal self-experiment.

The central question is no longer only whether a system crystal can support regrowth.
It is whether protocol crystals can also shape runtime layer routing.

The three-crystal set used here is:
- methodology crystal
- retrieval-routing crystal
- system crystal

## Experimental Question

Can protocol crystals shape not only how an agent enters system structure,
but also how it routes between memory, experience, crystal, and transfer layers at runtime?

---

## Main Observation

### 1. The retrieval-routing crystal changed the interpretation focus of the system crystal
With only methodology + system,
the main effect was on entry stance and regrowth discipline.

With methodology + retrieval-routing + system,
the model read the system crystal differently.
It no longer focused only on:
- what modules the system has
- how architecture should regrow

It also focused on:
- how runtime tasks should select retrieval layers
- when crystal or transfer material should remain gated
- how packaging order changes the reasoning trajectory

This suggests the retrieval-routing crystal is not merely content support.
It changes how system structure is interpreted under runtime conditions.

---

### 2. Protocol crystals now extend into runtime orchestration
The methodology crystal already suggested that protocol crystals can shape entry into a system crystal.

This experiment shows something stronger:
- protocol crystals can also shape runtime orchestration
- they can influence task-mode selection
- they can influence layer permissions
- they can influence packaging order

This expands the crystal framework beyond regrowth and transfer.

---

### 3. Runtime routing should be mode-first, not similarity-first
A strong outcome from the run is the clearer preference for:
- first classify task/runtime intent
- then select the correct retrieval layer(s)
- then apply permission boundaries
- then package retrieved material in a layer-aware order
- only then build the runtime context

This is directly opposed to a flat top-k retrieval assumption.

---

### 4. The crystal set is showing clearer internal role differentiation
At this stage the working distinction is becoming:

#### Protocol crystals
- methodology crystal
- retrieval-routing crystal

#### Content crystals
- system-memory crystal

This remains an experimental working distinction,
but it now has clearer support from runtime behavior.

---

## Main Working Conclusion

The three-crystal runtime-routing experiment suggests that protocol crystals are not limited to regrowth discipline.
They also shape runtime access and routing across layered knowledge structures.

This means the crystal framework is evolving from:
- knowledge transfer objects

toward:
- runtime protocol architecture

---

## Current Consequences

This result directly supports continued work on:
- retrieval-routing crystal as a protocol crystal
- crystal-type distinctions such as protocol vs content
- crystal-set schema with order and dependency fields
- runtime prompt construction as layered routing rather than flat retrieval

---

## Current Risk

This is still an early self-experiment.
What is supported:
- protocol crystals can shape runtime routing logic
- layered routing looks more faithful than flat retrieval

What is not yet fully proven:
- that all models respond equally well to the routing crystal
- that the current routing crystal is the right long-term structure
- that the same ordering will generalize to larger crystal sets

---

## One-Sentence Summary

The first three-crystal self-experiment suggests that protocol crystals can shape both entry into system structure and runtime layer routing, pushing the crystal framework toward a full runtime protocol architecture.
