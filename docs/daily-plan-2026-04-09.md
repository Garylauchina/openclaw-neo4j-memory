### Issue #36: 实体合并逻辑诊断

**问题：** 冥思报告中实体合并持续为 0

**可能原因：**
1. 图谱中确实没有重复实体（好）
2. 合并逻辑有问题，未触发（坏）

**诊断步骤：**
1. 运行 Neo4j 诊断查询（查找潜在重复实体）
2. 检查 `get_nodes_to_merge_with_priority()` 逻辑
3. 添加诊断日志

**验收标准：**
- [ ] 输出潜在重复实体列表（或确认无重复）
- [ ] 确认合并逻辑无 bug
- [ ] 添加诊断日志

**链接：** https://github.com/Garylauchina/openclaw-neo4j-memory/issues/36

### Issue #37: 冥思成本上限保护

**问题：** 随着图谱增长，LLM 调用成本可能失控

**功能要求：**
1. 单次冥思预算上限（`max_meditation_budget_per_run`）
2. 最大 LLM 调用次数（`max_llm_calls_per_run`）
3. 预算分配策略（6 个步骤的预算占比）
4. 降级策略（警告→跳过→停止）

**验收标准：**
- [ ] 配置项生效
- [ ] 预算超支时触发降级
- [ ] 日志包含成本监控信息
- [ ] 测试超支场景

## 🎯 开发流程

### 1. 认领任务
在 GitHub Issue 下评论认领，或直接在本地开始开发

### 2. 开发
```bash
# 创建分支
git checkout -b fix/issue-#36-entity-merge-diagnosis

# 编码并本地测试
python -c "import ast; ast.parse(open('meditation_memory/graph_store.py').read())"

# 提交
git add .
git commit -m "fix: 实体合并逻辑诊断 (Issue #36)"
```

### 3. 提交 PR
```bash
git push origin fix/issue-#36-entity-merge-diagnosis
gh pr create --title "fix: 实体合并逻辑诊断" --body "Fixes #36"
```

### 4. 审核
技术负责人会在 24 小时内审核并回复

### 5. 合并
审核通过后合并到 main 分支
